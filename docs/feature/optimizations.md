# Mapred 修复与架构优化说明

**日期**: 2026-03-18

本项目在多进程并发与文本流处理的基础上，进行了全面的修复和架构级的优化，以解决原代码在生产环境（尤其是处理极大数据流时）暴露出的潜在风险和瓶颈。主要优化点如下：

## 1. 彻底解决 OOM（内存爆炸）与引入基于水位的背压机制 (Backpressure)

* **问题背景**：原实现中，子进程写入事件会在循环中持续触发主标准输入 (`STDIN`) 的读取。由于主输入速度远大于某些慢速 mapper 的处理速度，当处理 GB 级以上的大文件时，主程序会将过量数据直接读取到内存缓存中，迅速导致 Out-Of-Memory (OOM) 崩溃。此外，对于完全没有换行符的恶意二进制长数据，也会引发单个 Buffer 无限扩张。
* **架构优化**：
    * **硬性 Buffer 限制** (`iostream.c`)：为 `try_read_more` 增加了单行最大 64MB 的上限防护，超限即返回 `E_ERROR` 拒绝读取。
    * **高低水位线机制** (`thread.c`)：重构了事件循环中的读取分发策略。为每个子进程配置独立的接收缓冲。当任何一个子进程积压的未处理数据**超过高水位线 (High Water Mark, 4MB)** 时，主输入流事件 (`stdin_ev`) 将被 `event_del` 自动摘除并挂起，暂停文件读取。
    * 当消费较慢的子进程将其积压数据**消化并降至低水位线 (Low Water Mark, 1MB)** 以下时，主输入流的监听事件会被 `event_add` 重新激活。从而真正实现了大文件处理时的零内存泄漏。

## 2. 解决 STDIN 阻塞卡死问题与事件重构

* **问题背景**：原代码未对 `STDIN_FILENO` 设置非阻塞。这导致 `libevent` 事件循环中，若调用 `read()` 恰巧没有数据可读时，会发生同步阻塞，不仅挂起事件循环本身，也导致多路复用的初衷失效。如果直接对其设置非阻塞（Non-blocking），又会因为旧的事件机制（只要可写就触发写事件）不断空转，导致 100% CPU 占用率的死循环。
* **架构优化**：
    * 颠覆了原来的“在写事件中顺便读”的设计，彻底拆分为 **独立的读事件监听器 (`EV_READ`)** 和 **按需触发的写事件监听器 (`EV_WRITE`)**。
    * 主线程注册 `stdin_ev` 专门监听 `STDIN_FILENO`。只有真正 `EV_READ` 触发并成功分配了数据行后，才会去激活 (`event_add`) 对应的子进程写事件。实现了真正的、优雅的**零阻塞和零空转的高性能异步分发**。

## 3. 实现基于 Round-Robin 的公平调度分发

* **问题背景**：因为原版是多个子进程竞争共享的 `wet->stream` 状态机，往往出现“抢到执行权的进程（或者消费快的进程）持续占用总线”，造成部分子进程**任务饥饿**或分配极其不均匀。
* **架构优化**：通过上述主读事件(`stdin_handler`)统揽分发大权后，程序现在采用完美的 **Round-Robin 轮询调度策略**：接收到文本行时，依次分配给下一个子进程的缓冲区中，确保了多核多进程负载的绝对均衡。

## 4. 修复信号竞争引起的滥杀进程隐患

* **问题背景**：在 `main.c` 的 `sig_handler`（用于处理 `SIGINT` 和 `SIGQUIT`）中，原程序通过遍历全局的 `pids` 数组发送 `kill` 信号，试图做到父子进程同生共死。但在启动初期如果数组未被完全初始化赋值，可能会向垃圾内存值（通常为极大的随机数，或者 `0` 指代当前进程组）误发致命信号，引发系统安全性问题。
* **修复动作**：
    * 将全局 `pids` 数组显式清零初始化 `{0}`。
    * 在执行遍历时，强制判断 `if (pids[i] > 0)` 确认子进程号有效后，再实施信号查杀，杜绝了初始化期的竞争危害。

## 5. 增强容错与 Fail-Fast 机制

* **问题背景**：虽然原版通过忽略 `SIGPIPE` 避免了子进程非正常死亡导致的管道断裂崩溃，但这会造成当用户指定的命令本身错误（如找不到 `awk`，或脚本存在严重 bug 在第一行就全部退出）时，主进程依然盲目执行并在 `wait_children` 中傻傻等待或直接把数据丢进黑洞，没有明显报错提醒。
* **修复动作**：修改 `os.c` 中的 `wait_children()` 逻辑。现在一旦主进程收集到任何子进程发生了**异常退出（被信号杀死）**或**返回非 0 失败错误码**，主程序会立刻停止等待并触发 `exit(EXIT_FAILURE)`。这为自动化部署流水线和 Pipeline 脚本组合提供了符合预期的 Fail-Fast 响应。
